The current code makes the working directory matter more than it should.

`relaymd` resolves config in this order: `RELAYMD_CONFIG` if set, otherwise `~/.config/relaymd/config.yaml`, then `./relaymd-config.yaml` in the current working directory, with the cwd file overriding the home file ([settings_sources.py](/depot/plow/data/pballane/folate-alpha-beta/relaymd/src/relaymd/settings_sources.py:13), [test_config.py](/depot/plow/data/pballane/folate-alpha-beta/relaymd/tests/orchestrator/test_config.py:109), [test_config.py](/depot/plow/data/pballane/folate-alpha-beta/relaymd/tests/cli/test_config.py:84)). The default DB URL is relative, `sqlite+aiosqlite:///./relaymd.db`, so the SQLite file lands in whatever directory you launch the orchestrator from unless your config overrides it with an absolute path ([orchestrator/config.py](/depot/plow/data/pballane/folate-alpha-beta/relaymd/src/relaymd/orchestrator/config.py:107)). That is why running from `test/` changes behavior: it picks up [test/relaymd-config.yaml](/depot/plow/data/pballane/folate-alpha-beta/relaymd/test/relaymd-config.yaml:1) and, unless you set `database_url`, will create `test/relaymd.db`.

**Where each thing needs to live today**

- Orchestrator: if you rely on `./relaymd-config.yaml`, run it from the directory that contains the config you want. Right now that means `test/` if you want the test-local config to apply automatically.
- DB file: wherever `database_url` points. If you do not set it, it becomes `./relaymd.db` in the orchestrator’s cwd.
- Job files to submit: anywhere on local disk. The CLI just takes the directory you pass, checks that it contains `relaymd-worker.json` or `relaymd-worker.toml` unless you use `--command`, tars it up, and uploads it. They do not need to be in the orchestrator directory at all ([submit.py](/depot/plow/data/pballane/folate-alpha-beta/relaymd/src/relaymd/cli/commands/submit.py:23)).
- React frontend: build it from `frontend/`. For normal use you do not run it as a separate long-lived service; the orchestrator serves the built assets from `frontend/dist`. `npm run dev` exists, but there is no Vite proxy config here, so that is not the main operational path ([frontend/package.json](/depot/plow/data/pballane/folate-alpha-beta/relaymd/frontend/package.json:1)).

**Better design**

The better model is: one global RelayMD control-plane config, one global orchestrator/frontend process, many project-local job bundles.

I would set it up like this:

1. Put the orchestrator config in `~/.config/relaymd/config.yaml` or another absolute path, and set an absolute `database_url`, for example:
```yaml
database_url: sqlite+aiosqlite:////home/you/.local/share/relaymd/relaymd.db
orchestrator_url: http://127.0.0.1:36158
log_directory: /home/you/.local/state/relaymd/logs
infisical_token: ""
```

2. Export `RELAYMD_CONFIG` once in your shell startup, or source one small env file once per shell:
```bash
export RELAYMD_CONFIG="$HOME/.config/relaymd/config.yaml"
export INFISICAL_TOKEN=...
```

3. Start the orchestrator from anywhere; cwd should stop mattering once config and DB paths are absolute:
```bash
cd /depot/plow/data/pballane/folate-alpha-beta/relaymd
source venv/bin/activate
relaymd orchestrator up
```

4. Keep project job bundles inside each project repo, for example:
- `/path/to/project-a/relaymd/job1/`
- `/path/to/project-b/relaymd/job2/`

Each bundle directory should contain its worker config plus input files. Then submit with an explicit path:
```bash
source /depot/plow/data/pballane/folate-alpha-beta/relaymd/venv/bin/activate
relaymd submit /path/to/project-a/relaymd/job1 --title "project-a job1"
```

5. Treat the React frontend as part of the orchestrator deployment, not as a separate app you start in another terminal. Build it when the frontend changes, then restart the orchestrator:
```bash
cd /depot/plow/data/pballane/folate-alpha-beta/relaymd/frontend
npm --cache ./.npm run build
```

That gives you the right separation:
- RelayMD instance state is global.
- Project input bundles are local to each project.
- The orchestrator does not depend on being launched from `test/`.
- The frontend is not a third manually-managed runtime.
