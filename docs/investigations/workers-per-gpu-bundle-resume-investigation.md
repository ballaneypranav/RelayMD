# Investigation: Workers-per-GPU Changes Across Resume

## Summary

This investigation captures an operational edge case found on 2026-05-17 while
running AToM/RelayMD production bundles.

A set of jobs was submitted with `workers-per-gpu=1`, but the intended runtime
fanout was `workers-per-gpu=4`. The generated input bundle encoded that value in
`run.sh` as `WORKERS_PER_GPU=1`. Once the job was submitted, RelayMD uploaded the
bundle to object storage and future resume/requeue attempts continued using the
same uploaded input bundle.

The key finding is:

- RelayMD correctly treats input bundles as immutable job input, but it does not
  currently provide a supported way to start a new job from an existing job's
  checkpoint while using a newly generated input bundle.

## User-Reported Scenario

The concrete bundle inspected was:

`/scratch/gilbreth/pballane/folate-alpha-beta/FR_ATM-LSFE-01-S3C-R1-THF/output/07-production-relaymd-bundles/5THG_APTX/run.sh`

The generated script contained:

```bash
WORKERS_PER_GPU=1
```

The adjacent worker config only instructed RelayMD to run the script:

```json
{
  "command": ["bash", "run.sh"]
}
```

Because `workers-per-gpu` is generated by the AToM input bundle preparation
workflow, it is not a RelayMD domain parameter. RelayMD should not grow a generic
`relaymd jobs requeue --set workers-per-gpu=4` flag for this case.

## Current RelayMD Behavior

RelayMD submission currently works as follows:

1. The CLI creates a job ID.
2. The CLI archives the input directory into `bundle.tar.gz`.
3. The CLI uploads that archive to `jobs/<job_id>/input/bundle.tar.gz`.
4. The orchestrator stores `input_bundle_path` on the job.
5. Workers download that bundle on each execution segment.
6. Requeue clones copy the original job's `input_bundle_path`.

Relevant code paths:

- `src/relaymd/cli/commands/submit.py` creates and uploads the immutable bundle.
- `src/relaymd/cli/services/submit_service.py` registers jobs with an
  `input_bundle_path`.
- `src/relaymd/orchestrator/services/job_transitions.py` copies
  `input_bundle_path` in `build_requeue_clone()`.
- `packages/relaymd-worker/src/relaymd/worker/main.py` downloads and executes the
  configured bundle.

Relevant documented invariant:

- `docs/architecture.md` states that input bundles are immutable and should not be
  overwritten. If input needs to change, create a new job.

## Operational Workaround Considered

One emergency workaround is to overwrite the existing object storage key
`jobs/<job_id>/input/bundle.tar.gz` with a patched bundle. That would affect the
next worker segment because it downloads the bundle fresh.

This was rejected as the product direction because it violates RelayMD's
immutability invariant and makes the historical job record ambiguous: the same
job ID would appear to have run a bundle whose contents changed after submission.

For the immediate production need, a fresh job with a correctly generated bundle
was submitted instead.

## Preferred Product Direction

The better model is: create a new job ID from a new bundle while resuming from an
old job ID's checkpoint.

This preserves the boundary between RelayMD and workload-specific bundle
generation:

- The AToM bundle preparation script owns `workers-per-gpu` and renders `run.sh`.
- RelayMD owns job IDs, immutable bundle references, checkpoint references,
  worker assignment, and job history.

A proposed operator flow:

```bash
relaymd submit ./new-bundle-dir \
  --title "5THG_APTX resume mps4" \
  --resume-from-job <old-job-id> \
  --cluster gilbreth-standby-a100-40gb-mps4
```

Alternative spelling:

```bash
relaymd jobs resume-as-new <old-job-id> ./new-bundle-dir \
  --title "5THG_APTX resume mps4" \
  --cluster gilbreth-standby-a100-40gb-mps4
```

Expected internal behavior:

1. Upload the new bundle to `jobs/<new-job-id>/input/bundle.tar.gz`.
2. Create a new queued job with the new `input_bundle_path`.
3. Copy `latest_checkpoint_manifest_path` and `last_checkpoint_at` from the old
   job.
4. Apply new cluster affinity supplied by the operator.
5. Record job history linking the new job to the old job and checkpoint source.

## Design Notes

Plain `requeue` should continue to mean "run the same logical job again with the
same bundle/runtime assumptions." If the bundle changes, even if the only change
is a generated `run.sh` parameter, the operation should create a distinct job ID.

This also keeps debugging and audit trails straightforward:

- Old job ID: old bundle, old runtime choices, old cluster affinity.
- New job ID: new bundle, new runtime choices, checkpoint inherited from old job.

## Follow-Up Work

Implement a supported resume-as-new flow that can create a new job from a new
bundle and an existing job's checkpoint metadata. Add API, CLI, tests, and job
history events for this operation.
